# CACP 协议宣介动画设计文档

## 项目信息

- **日期：** 2026-04-28
- **项目：** CACP（Collaborative Agent Communication Protocol）协议宣介动画
- **使用场景：** README 置顶 / 项目首页展示
- **目标时长：** 约 150-160 秒（2 分 30 秒 - 2 分 40 秒）

## 目标受众

- **主要：** 模型厂商（OpenAI、Anthropic、Google 等）的技术决策者
- **次要：** AI 工具开发者、开源社区贡献者
- **再次：** 对多人 AI 协作感兴趣的普通用户

## 核心主张

CACP 不是一个产品，而是一个**开放通信协议**——类比 HTTP 之于互联网、SMTP 之于邮件。它定义了"多人 + AI"的协作通信标准，旨在改变当前 AI 只能一对一交互的范式局限。

**当前状态：** 协议已就绪（v0.2），参考实现已完成，Agent 生态正在建设中，需要社区共建。

## 第一幕：AI 单人交互的范式困境（~40 秒）

### 1.1 开场（0:00-0:10）

- **画面：** 纯黑背景，白色文字淡入
- **文案：** "当前，所有 AI 交互都是一对一"
- **视觉：** 文字下方快速闪过各种 AI 聊天界面（ChatGPT、Claude、Copilot 等）的模糊残影
- **音乐：** 低沉、略带机械感的电子音效

### 1.2 场景展示（0:10-0:25）

- **画面：** 三个并排的屏幕，每个屏幕展示一个独立场景
  - 左：产品经理独自问 AI "如何设计登录流程"
  - 中：设计师独自问 AI "登录页用什么配色"
  - 右：开发者独自问 AI "OAuth 怎么实现"
- **视觉：** 三个屏幕之间有虚线分隔，暗示信息不互通
- **文案（旁白）：** "同一个问题，三个人各自问各自的 AI"

### 1.3 痛点揭示（0:25-0:40）

- **画面：** 三个屏幕各自显示不同的 AI 回答，答案不一致甚至矛盾
- **视觉：** 屏幕逐渐变灰，画面中央出现红色警示文字
- **文案（逐个浮现）：**
  - "信息孤岛"
  - "上下文断裂"
  - "结论碎片化"
- **收尾：** 画面变暗，白色大字："AI 时代的人机交互，还停留在单人聊天模式"

## 第二幕：CACP 协议——新的协作范式（~65 秒）

### 2.1 转折（0:40-0:50）

- **画面：** 黑底，文字逐字打出
- **文案：** "如果 AI 能像加入 Slack 频道一样，加入人类的协作空间呢？"
- **音乐：** 音调开始上扬，出现希望的音色

### 2.2 协议定位（0:50-1:05）

- **画面：** CACP Logo 淡入，然后展开为协议分层架构图
- **视觉：** 三层架构（参考 `docs/cacp-concept.svg`）
  - **参与者层：** 房主、成员、观察者
  - **共享协调层：** 协作房间（事件日志 + 消息收集 + 角色权限）
  - **AI 执行层：** 共享本地 Agent
- **文案（旁白）：** "CACP 不是另一个聊天工具，它是一个开放通信协议"
- **标签浮现：** "正如 HTTP 统一了网页通信，CACP 统一了多人与 AI 的协作通信"

### 2.3 事件驱动架构（1:05-1:20）

- **画面：** 展示事件流动画
- **视觉：** 事件卡片从左到右流动
  - `message.created` → `ai.collection.started` → `ai.collection.submitted` → `agent.turn.requested` → `agent.turn.completed`
- **文案（旁白）：** "所有状态变化都是不可变事件，任何客户端都能从事件流重建完整状态"
- **标签：** "事件溯源 · 状态一致 · 可审计"

### 2.4 范式对比（1:20-1:35）

- **画面：** 左右分屏对比
  - **左侧（传统）：** "人 → AI" 单向箭头，灰色
  - **右侧（CACP）：** 多人与 AI 之间形成网状连接，彩色
- **文案（旁白）：** "传统模式下，AI 只服务一个人。CACP 让 AI 成为团队协作的一员"

### 2.5 AI Flow Control 演示（1:35-1:55）

- **画面：** 展示 CACP 房间界面动画
- **视觉流程：**
  1. 房主点击 "Start collecting answers"
  2. 成员 A 发送消息，标记 "Queued for AI"
  3. 成员 B 发送消息，同样标记
  4. 房主点击 "Submit collected answers"
  5. 两条消息合并为一次 Agent 输入
  6. Agent 给出综合分析回复
- **文案（旁白）：** "房主控制 AI 流转节奏，收集多人回答后再统一提交，避免 AI 被碎片化信息淹没"

## 视觉风格

- **整体：** 简洁科技风，偏向协议/标准文档的严肃感，而非消费产品的活泼感
- **配色：**
  - 主色：深蓝 `#1a1a2e`（背景）、亮蓝 `#2563eb`（协议层）
  - 强调色：橙 `#c76b2a`（控制流）、紫 `#7c3aed`（AI 输入）
  - 辅助：灰白 `#faf8f5`（卡片背景）
- **字体：** system-ui / sans-serif，标题粗体，正文常规
- **动画：** 平滑的淡入淡出、线条绘制、节点高亮，避免花哨特效
- **图标风格：** 扁平化、线条简洁，与 `docs/cacp-concept.svg` 一致

## 音频建议

- **背景音乐：** 极简电子氛围音乐，节奏随叙事推进
  - 第一幕：低沉、略带压迫感
  - 第二幕：音调上扬，出现明亮合成器音色
  - 第三幕：开放、轻盈
  - 第四幕：宏大、收尾
- **旁白：** 可选，中文或英文取决于目标受众。如果无旁白，需确保文案文字足够清晰传达信息
- **音效：** 节点连接时的轻微"叮"声、事件流动的 subtle tech sound

## 技术实现

- **工具：** HyperFrames（HTML-based video composition）
- **素材：**
  - `docs/cacp-concept.svg` 作为协议架构图素材
  - 模拟的 CACP Web UI 界面（可用简化版 HTML/CSS 实现）
  - 事件流动画（CSS/JS 动画）
- **输出：** MP4 / WebM，建议 1920x1080 或 1280x720，支持循环播放

## 成功标准

1. 观众在 30 秒内理解"CACP 是一个协议，不是产品"
2. 观众能记住"改变 AI 单人交互范式"这个核心主张
3. 模型厂商观众产生"我们可以适配这个协议"的兴趣
4. 开发者观众产生"我可以贡献 Agent 适配器"的参与感
